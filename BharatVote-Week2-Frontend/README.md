# BharatVote - Week 2 Frontend: Contract Integration & Type Safety

## ğŸ“‹ Purpose

This is the **Week 2 frontend implementation** of the BharatVote blockchain voting system. It builds upon Week 1's wallet connection foundation by integrating with the deployed smart contract, enabling the frontend to read blockchain state (admin status, election phase, candidates) and display role-based UI. - **Role-based UI**: Different interface elements for admin vs. voters

## ğŸ¯ Key Implementation Details

### 1. Contract Instance in useWallet Hook

**Decision**: Create contract instance in `useWallet` hook rather than in individual components.

**Benefits**:
- **Single source of truth**: One contract instance shared across app
- **Event listeners persist**: Don't need to recreate on every render
- **Automatic updates**: Recreates on account/network change
- **Consistent provider**: Uses same provider/signer throughout

**Alternative Considered**: Creating instances on-demand in components (rejected due to complexity and event listener management).

### 2. Type-Safe Contract Interface

**Decision**: Define custom `BharatVoteContract` interface extending `BaseContract`.

**Benefits**:
- **Autocomplete**: IDE suggests all contract methods
- **Compile-time safety**: Catches typos and wrong method signatures
- **Better DX**: Type hints for return values
- **Prevents runtime errors**: Invalid calls caught during development

### 3. Direct RPC Provider for State Reading

**Decision**: Use direct `JsonRpcProvider` for reading contract state in `App.tsx`.

**Why?** MetaMask provider sometimes caches old blockchain state. Direct provider ensures we always get latest state.

```typescript
const rpcProvider = new ethers.JsonRpcProvider('http://127.0.0.1:8545');
const contractWithRPC = new ethers.Contract(
  contractAddress,
  contractInterface,
  rpcProvider
);
```

**Note**: Still use MetaMask provider for transactions (needs signing).

### 4. Circuit Breaker Pattern for RPC Calls

**Decision**: Implement retry logic with exponential backoff for RPC calls.

**Why?** Hardhat node sometimes takes a moment to process blocks. Retry logic handles transient errors gracefully.

```typescript
const retryWithDelay = async (fn, retries = 4, initialDelay = 800) => {
  let delay = initialDelay;
  for (let i = 0; i < retries; i++) {
    try {
      return await fn();
    } catch (err) {
      if (isCircuitBreakerError(err) && i < retries - 1) {
        await new Promise(res => setTimeout(res, delay));
        delay = Math.floor(delay * 1.5);
        continue;
      }
      throw err;
    }
  }
};
```

### 5. Event Listeners for Real-Time Updates

**Decision**: Listen for contract events instead of polling.

**Benefits**:
- **Efficient**: No unnecessary RPC calls
- **Instant updates**: UI updates immediately when events fire
- **Better UX**: Users see changes in real-time
- **Reduced load**: Less stress on RPC provider

**Events Listened To**:
- `PhaseChanged`: Updates phase badge
- `CandidateAdded`: Refreshes candidate list
- `CandidateRemoved`: Refreshes candidate list

### 6. Admin Detection Logic

**Implementation**: Compare `contract.admin()` result with connected account address.

```typescript
const adminAddress = await contract.admin();
const isAdmin = adminAddress.toLowerCase() === account.toLowerCase();
setIsAdmin(isAdmin);
```

**Why blockchain as source of truth?** Contract state is authoritative. No hardcoded addresses needed.

## ğŸ› Troubleshooting

### "Contract not available" Error

**Symptoms**: App shows error about contract not being found.

**Solutions**:
1. **Check Hardhat node is running**: `npm run node` in backend directory
2. **Check contract is deployed**: Run `npm run verify` in backend directory
3. **Check contract address**: Verify `src/contracts/BharatVote.json` has correct address
4. **Check network**: Ensure MetaMask is on Localhost 8545 network

### Admin Detection Not Working

**Symptoms**: Connected as admin but admin badge doesn't show.

**Solutions**:
1. **Verify deployer account**: Ensure you're using the account that deployed the contract
2. **Check console logs**: Open browser console, look for admin address logs
3. **Verify contract admin**: Run in Hardhat console: `await contract.admin()`
4. **Check address comparison**: Ensure case-insensitive comparison is working

### Phase Not Updating

**Symptoms**: Phase badge shows old phase after calling `startReveal()`.

**Solutions**:
1. **Check event listeners**: Verify listeners are registered (check console logs)
2. **Check contract phase**: Run in Hardhat console: `await contract.phase()`
3. **Refresh page**: Sometimes event listeners need refresh after connection
4. **Check RPC provider**: Verify direct provider is working correctly

### "Provider not found" Error

**Symptoms**: Cannot connect wallet, error about missing provider.

**Solutions**:
1. **Install MetaMask**: Download and install MetaMask browser extension
2. **Refresh page**: After installing MetaMask, refresh the page
3. **Check MetaMask is unlocked**: Ensure MetaMask is unlocked and ready
4. **Check browser compatibility**: Use Chrome, Firefox, or Brave

### Candidates Not Showing

**Symptoms**: No candidates displayed even though they exist in contract.

**Solutions**:
1. **Check contract state**: Run `await contract.getCandidates()` in Hardhat console
2. **Check event listeners**: Verify `CandidateAdded` listener is registered
3. **Manual refresh**: Try disconnecting and reconnecting wallet
4. **Check ABI**: Ensure `BharatVote.json` has correct ABI with `getCandidates` method

### Circuit Breaker Errors

**Symptoms**: Frequent "circuit breaker" errors in console.

**Solutions**:
1. **Check Hardhat node**: Ensure node is running and responsive
2. **Increase retry delay**: Modify retry logic in `App.tsx` if needed
3. **Check RPC URL**: Verify `http://127.0.0.1:8545` is correct
4. **Restart Hardhat node**: Sometimes node needs restart to clear state

## ğŸ“š Key Concepts Explained

### **Ethers.js v6**
Modern Ethereum library for interacting with blockchain. Week 2 uses:
- `BrowserProvider`: MetaMask provider
- `JsonRpcProvider`: Direct RPC connection
- `Contract`: Contract instance for method calls
- `ContractTransaction`: Return type for state-changing functions

### **TypeScript Type Assertions**
```typescript
const contract = new ethers.Contract(...) as unknown as BharatVoteContract;
```
We use type assertion because Ethers.js's `Contract` type is generic. Our custom interface provides specific types for our contract.

### **React Hooks**
- `useState`: Manages component state (admin, phase, candidates)
- `useEffect`: Side effects (event listeners, contract reads)
- `useCallback`: Memoized functions (error handling)
- `useRef`: Persistent values across renders (connection state)

### **Event-Driven Architecture**
Instead of polling the contract every few seconds, we listen for events. When contract emits an event, our listeners fire and update UI instantly.

### **Circuit Breaker Pattern**
A resilience pattern that prevents cascading failures. If an operation fails, we retry with exponential backoff instead of failing immediately.

### **Role-Based Access Control (RBAC)**
UI adapts based on user role (admin vs. voter). Admin sees management options, voters see voting options (Week 3+).
